# Spec: Согласие на отображение имени на табло (киоск)

**Дата:** 2026-05-27  
**Проект:** eque  
**Статус:** approved

---

## Контекст

Электронное табло (`/board/:slug`) публично отображает ФИО пациентов в холле больницы, что потенциально нарушает медицинскую тайну. ~50% пациентов — геронтологические: не могут запомнить номер, со старыми телефонами или без навыков работы со смартфоном. Физический талон обязателен.

Инфраструктура печати:
- Киоски: термопринтеры установлены физически (в коде не интегрированы — отдельная задача)
- Регистратура: обычные A4-принтеры

Данная фича решает проблему через механизм добровольного согласия пациента при записи через киоск.

---

## Решение

После заполнения ФИО в киоске появляется отдельный экран с вопросом о согласии на отображение имени на табло. На основе ответа запись сохраняется с флагом `displayConsent`. Табло рендерит имя или номер талона в зависимости от этого флага.

---

## Секция 1: Данные и бизнес-логика

### Новое поле

```prisma
model QueueEntry {
  // ...существующие поля...
  displayConsent Boolean @default(true)
}
```

**Правила:**
- Захватывается только при создании через киоск (`kiosk.addToQueue`)
- Записи через регистратора, колл-центр, врача → всегда `true` (дефолт, поведение табло не меняется)
- Неизменяемо после создания

### Логика display.router.ts

```
displayConsent = true  → возвращать patientFirstName + patientLastName (как сейчас)
displayConsent = false → patientFirstName = null, patientLastName = null
```

`queueNumber` передаётся всегда в обоих случаях.

### Логика табло (frontend)

```
если firstName != null → рендерить "Алексей Ив. → каб. 3"  (как сейчас)
если firstName == null → рендерить "№{queueNumber} → каб. 3"
```

TTS (`useCallNotifications.ts`):
```
если firstName != null → произносить ФИО (как сейчас)
если firstName == null → произносить "Номер {queueNumber}, кабинет {cabinetNumber}"
```

## Секция 2: Киоск UI — новый экран согласия

### Изменение флоу

**Было:** `'welcome'` → `'entry'` → `addMutation` → `'confirm'`  
**Стало:** `'welcome'` → `'entry'` → `'consent'` → `addMutation` → `'confirm'`

Новый state `'consent'` вставляется между `'entry'` и вызовом `addMutation`.

### Триггер перехода

Кнопка «✓ Келесі / Далее» на последнем поле (`middleName`) больше не вызывает `addMutation` напрямую — переходит на экран `'consent'`.

### Экран consent

Полноэкранный, в стиле остального киоска (тёмный градиент `#00685B → #004d44`, без иконок):

```
[Логотип]

┌─────────────────────────────────┐
│  Сіздің деректеріңіз / Данные   │
│  Иванов Алексей                 │
│  Петрович                       │
└─────────────────────────────────┘

Атыңызды ақпараттық тақтада
көрсетуге рұқсат бересіз бе?

Разрешаете отображать ваше имя
на информационном табло?

┌─────────────────────────────────┐  ← фон #B39168
│  ✓ Иә, келісемін / Да          │
│  На табло: «Алексей Ив.»       │
└─────────────────────────────────┘

┌─────────────────────────────────┐  ← фон rgba(255,255,255,.12)
│  ✗ Жоқ / Нет                   │
│  На табло: только «№5»         │
└─────────────────────────────────┘

                        ← Артқа / Назад
```

**Детали:**
- Имя пациента в карточке берётся из уже заполненного `fields` (не делает новый запрос)
- Подсказка на кнопке «Нет»: `"На табло: только порядковый номер"` — без конкретного `№X` (номер неизвестен до создания записи)
- Нажатие любой кнопки вызывает `addMutation` с соответствующим `displayConsent: true/false`
- Кнопки блокируются во время `addMutation.isPending`

### Обработка ошибок на экране consent

Если `addMutation` завершилась ошибкой:
- Остаёмся на экране `'consent'`
- Показываем `addMutation.error.message` (аналогично текущему поведению на `'entry'`)
- Кнопки снова активны

### Кнопка «Назад»

Маленькая текстовая кнопка внизу экрана: `← Артқа / Назад`  
Действие: `setScreen('entry')`, `addMutation.reset()`

## Секция 3: Табло — изменения компонентов

### Затрагиваемые файлы

| Файл | Изменение |
|---|---|
| `board/ActiveCallsPanel.tsx` | `firstName ?? null` → рендер `№{queueNumber}` если null |
| `board/QueuePanel.tsx` | аналогично |
| `board/CallOverlay.tsx` | аналогично |
| `board/useCallNotifications.ts` | TTS: если нет имени → "Номер X, кабинет Y" |
| `display.router.ts` | условно не передавать firstName/lastName |

### Правило рендера (все 3 компонента)

```typescript
// Вместо: `${entry.patientFirstName} ${entry.patientLastName.slice(0,2)}.`
// Писать:
entry.patientFirstName
  ? `${entry.patientFirstName} ${entry.patientLastName.slice(0,2)}.`
  : `№${entry.queueNumber}`
```

## Секция 4: Граничные случаи

| Случай | Поведение |
|---|---|
| Лимит киоска исчерпан | Проверка на бэке при `addMutation`; экран `'consent'` появится, затем получит ошибку — отобразим её |
| Существующие записи в БД | Миграция: `@default(true)` — поведение табло не меняется |
| Записи не через киоск | `displayConsent = true` по умолчанию — поведение не меняется |
| `queueNumber` в подсказке кнопки «Нет» | Неизвестен до создания записи — показывать `№…` или убрать |

## Не входит в scope

- Интеграция термопринтера для печати талона (отдельная задача)
- Механизм согласия для регистраторского потока
- Юридическая формулировка согласия (согласовывается отдельно с юристом, placeholder в коде)
- Изменение текста согласия в настройках через UI
